## 왜 시작했나

채용 담당자, 기술 리더, 개발자 커뮤니티 독자 모두가 방문할 수 있는 개인 홈페이지가 필요했다.
블로그 + 프로젝트 포트폴리오 + 서비스 플랫폼을 하나로 묶되, **1주일 안에 배포 가능한 MVP**를 만드는 것이 목표였다.

AI 코딩 도구가 많이 나왔지만, 실제로 프로덕션 수준의 프로젝트를 처음부터 끝까지 AI와 함께 짓는 경험을 해보고 싶었다.
선택한 도구는 **Claude Code** — Anthropic이 만든 CLI 기반 AI 코딩 에이전트다.

---

## 기술 스택 결정

본격적인 코딩 전에 기술 스택을 먼저 고정했다.
변경 비용이 크기 때문에 초반에 명확히 정해두는 게 낫다.

| 영역 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | **Next.js 15 App Router** | 가장 익숙하고, Cloudflare 어댑터 지원 |
| 스타일링 | **Tailwind CSS v4** | v4의 CSS 변수 방식이 shadcn과 잘 맞음 |
| UI 컴포넌트 | **shadcn/ui** | 복사-붙여넣기 방식으로 의존성 최소화 |
| 콘텐츠 | **Velite + MDX** | Next.js 통합이 자연스럽고 타입 안전 |
| 배포 | **Cloudflare Pages + Workers** | 무료 티어, 엣지 성능, D1/R2 연동 |
| 패키지 매니저 | **pnpm** | 빠르고 디스크 효율적 |

## Claude Code 운영 규칙 설계

Claude Code를 단순한 자동완성 도구가 아닌 **일관된 협업자**로 쓰려면 규칙이 필요했다.
`CLAUDE.md`를 프로젝트 루트에 두고, 세부 규칙은 `docs/claude/` 아래에 분리했다.

```
CLAUDE.md                  ← 진입점 (파일 참조만)
docs/claude/
  project.md               ← 프로젝트 목적, 기술 스택, 출력 언어
  workflow.md              ← 작업 흐름, 커밋 규칙, 응답 형식
  architecture.md          ← 데이터 역할 분리, 폴더 구조, 금지 사항
  mistakes.md              ← 반복 실수 기록 (누적 관리)
```

**핵심 규칙 중 몇 가지:**
- 작업 시작 전 반드시 `techspec.md` → `plan.md` 순서로 읽는다
- 한 번에 하나의 기능 단위만 구현한다
- 코드 생성 전에 작업 계획을 먼저 설명한다
- 모든 설명과 커밋 메시지는 한국어로 작성한다

이 규칙 덕분에 Claude가 맥락을 잃지 않고 일관되게 작업할 수 있었다.

## Day 1 — 기본 구성 완성

Day 1은 총 4개의 커밋으로 진행됐다.

### Step 1 — shadcn/ui 초기화

```
pnpm dlx shadcn@latest init 기준으로 진행한다.
스타일은 과하지 않은 기본값을 사용한다.
CSS 변수 사용 방식을 우선한다.
```

**예상치 못한 이슈:**
shadcn CLI 최신 버전에서 `--base-color` 플래그가 없어졌다.
여러 번 시도 끝에 `-b radix -p nova` 조합으로 비대화형 실행에 성공했다.

```bash
pnpm dlx shadcn@latest init -y -f --css-variables -b radix -p nova
```

![shadcn/ui 공식 사이트](https://ui.shadcn.com/og.jpg)

**결과:**
- `components.json` 생성
- `src/lib/utils.ts` — `cn()` 유틸 함수
- `globals.css` — oklch 기반 CSS 변수 블록 (`:root` / `.dark`)

---

### Step 2 — ESLint 마이그레이션 (예상치 못한 이슈)

`pnpm lint` 실행 시 에러 발생:

```
Invalid project directory provided, no such directory: .../lint
```

**원인:** Next.js 16에서 `next lint` 명령이 **완전히 제거**됐다.
공식 문서를 확인해 공식 codemod를 실행했다.

```bash
npx @next/codemod@canary next-lint-to-eslint-cli . --force
```

codemod가 자동으로 `package.json`의 `"lint": "next lint"`를 `"lint": "eslint ."`로 바꿔줬다.
하지만 생성된 `eslint.config.mjs`에 ESM `.js` 확장자 누락, `eslint-config-next` 버전 불일치(15 → 16) 등 추가 이슈가 있었고, 이를 단계적으로 해결했다.

**배운 점:** 메이저 버전 업그레이드 시 공식 마이그레이션 가이드를 먼저 확인하는 게 가장 빠르다.

---

### Step 3 — 공통 레이아웃 컴포넌트

```
과한 디자인 작업은 하지 말고 공통 레이아웃의 최소 골격만 만든다
아직 모바일 메뉴 토글, 검색, 다크모드 스위치 같은 고급 기능은 넣지 않는다
```

생성한 컴포넌트:

```
src/components/layout/
  site-header.tsx    ← 로고 + 네비게이션 링크 5개
  site-footer.tsx    ← 저작권 + GitHub/소개 링크
  page-container.tsx ← max-w-4xl 공통 래퍼
```

**설계 결정 하나:** `PageContainer`를 `RootLayout`에서 전체 감싸지 않고, **각 페이지가 직접 사용**하는 방식을 선택했다.
나중에 풀너비 히어로 섹션 같은 예외 케이스를 유연하게 처리하기 위함이다.

![Tailwind CSS v4](https://tailwindcss.com/img/social-square.jpg)

### Step 4 — 페이지 라우트 스텁

네비게이션 링크가 404로 떨어지지 않도록 최소 스텁 페이지를 만들었다.

```
app/
  page.tsx              ← 홈 (기본 템플릿 제거)
  blog/page.tsx         ← 블로그 목록 스텁
  blog/[slug]/page.tsx  ← 블로그 상세 스텁
  projects/page.tsx     ← 프로젝트 목록 스텁
  showcase/page.tsx     ← AI Showcase 목록 스텁
  about/page.tsx        ← 소개 스텁
```

Next.js 15+에서는 동적 라우트의 `params`가 Promise로 바뀌었다.

```tsx
// Next.js 15+ 방식
export default async function BlogPostPage({ params }: Props) {
  const { slug } = await params;
  // ...
}
```

## Claude Code와 협업하며 느낀 것

**좋았던 점:**
- "코드 작성 전에 계획을 먼저 설명해라" 규칙이 효과적이었다. 방향이 잘못됐을 때 코드 한 줄 없이 수정할 수 있었다.
- 에러가 발생하면 공식 문서를 찾아서 근거 있는 해결책을 제시했다. 랜덤 추측이 아니었다.
- 커밋 메시지를 일관된 형식으로 작성해줘서 협업하는 느낌이 났다.

**아쉬웠던 점:**
- `pnpm preview` 재로드 이슈처럼, 환경 특이 케이스에서는 여러 번 시도가 필요했다.
- 토큰 컨텍스트가 길어지면 앞 내용을 잊는 경우가 있었다. `CLAUDE.md` 규칙 파일의 중요성을 다시 느꼈다.
