> 📌 **한 줄 요약:** 투명 PNG 한 장 헤더에 박는데 4시간 걸렸어요. AI가 빠른 게 아니라, 인간이 옳은 방향을 짚어줘야 빨라진다는 걸 다시 배운 날이에요.

안녕하세요, 고텍이예요! 오늘은 좀 특이한 글이에요. 보통은 "이거 직접 해봤더니 좋더라" 류의 후기를 쓰는데, 오늘은 **AI랑 4시간 동안 헛발질한 회고**예요.

작업 자체는 진짜 단순했어요. 내가 직접 만든 `logo.png` 한 장을 GoTechy 사이트 헤더에 라이트/다크 모드 모두 자연스럽게 박는 것. 보통 30분이면 끝날 일인데, AI가 자꾸 엉뚱한 데서 헤매고 나는 답답해서 같은 말을 다섯 번씩 반복하다 보니 **4시간**이 됐어요.

끝나고 나서 "도대체 어디서 잘못됐지?" 정리해봤더니, **AI 페어 프로그래밍에서 흔히 빠지는 5가지 함정 패턴**이 나오더라고요. 다음에 비슷한 상황 만나는 분들이 같은 시간 낭비 안 하시도록, 솔직하게 까볼게요. 같이 고!!

## 뻘짓 #1 — Read 도구의 시각화 함정

처음 문제는 여기서 시작됐어요.

Claude Code의 Read 도구로 PNG 파일을 열면, 투명 PNG라도 **흰 배경 위에 합성해서** 보여줘요. 그래서 내 `logo.png`(GoTechy 워드마크가 라이트 그레이로 디자인된 투명 PNG)를 AI가 Read로 열면, 흰 배경에 흐릿한 라이트 그레이 글자가 거의 안 보이는 상태로 인식돼요.

근데 저는 같은 파일을 Mac 다크 모드 뷰어에서 띄워서 캡처해서 보냈거든요. 어두운 배경 위에서는 똑같은 라이트 그레이 픽셀이 **흰색에 가까운 또렷한 GoTechy 글자**로 보여요.

**같은 픽셀, 같은 파일, 합성 배경만 다른 거였어요.**

근데 AI는 "이건 logo.png가 아니에요. 다른 로고가 캐시된 것 같아요"라고 단정. 저는 "야 그거 내가 클릭해서 띄운 파일 캡처한 거라고!" 같은 말을 다섯 번 반복했어요. ㅋㅋ

**교훈:** 도구의 시각화는 항상 **어딘가에 합성된 결과물**이에요. 같은 파일이라도 도구·뷰어마다 보여주는 방식이 다를 수 있다는 걸 의식해야 해요.

## 뻘짓 #2 — 광학 인지 vs 픽셀 데이터

다음 컴플레인은 이거였어요.

> "라이트 모드에서 G 아이콘이 파스텔 cyan/lavender로 떠 보이는데, 다크 모드는 진한 틸로 또렷해. 두 모드 색이 달라."

AI가 numpy로 두 PNG의 G 아이콘 영역 픽셀을 직접 비교했어요. 결과는:

```
G icon area diff: 0 pixels
alpha channel diff: 0 pixels
```

**완전히 동일한 픽셀**이에요. 그러면 왜 다르게 보일까? **흰 배경 위에서 같은 색은 더 밝고 채도 낮게 인지되고, 검은 배경 위에서는 더 진하고 채도 높게 인지되는 광학 효과** 때문이에요. 같은 회색을 흰 종이 위에 놓을 때와 검은 종이 위에 놓을 때 다르게 보이는 거랑 같아요.

근데 AI는 광학 문제를 PNG 변환으로 해결하려고 했어요. CSS `filter: saturate(1.18) brightness(0.88) contrast(1.06)`을 들이대고, 나중엔 PNG 자체의 RGB를 ×0.78로 어둡게 처리하는 변환본까지 만들었어요. 저는 "내 PNG 좀 그만 만져!!"라고 화내고… ㅎㅎ

**교훈:** 광학 인지 차이와 픽셀 데이터는 **다른 레이어의 문제**예요. 본질이 어디 있는지부터 명확히 하고 손대야 해요. 안 그러면 본질이 아닌 곳에서 시간 다 갑니다.

## 뻘짓 #3 — 줌인의 함정 (이거 진짜 컸어요)

여기가 제일 답답했던 부분이에요.

라이트 모드 헤더 좌상단에 "흰색 박스가 보인다"는 컴플레인이 나왔을 때, AI는:

1. BrandLogo 컴포넌트 봄 → background 없음 ✓
2. BrandLogo가 쓰는 PNG 봄 → 누키 완벽, 흰 픽셀 0개 ✓
3. `.brand-logo*` CSS 클래스 봄 → background 없음 ✓
4. next/image 서빙 결과 봄 → 완벽 투명 ✓
5. "원인 못 찾겠다" 😱

그 좁은 시야 안에서만 빙빙 돈 거예요. **진짜 범인은 한 단계 위, `layout.tsx`의 wrapper였어요:**

```css
/* src/app/globals.css */
.light .app-frame {
  background: var(--do-surface);  /* = #FFFFFF (순백색!) */
  border-radius: 1.5rem;
  margin: 0 0.75rem;
  overflow: clip;
  box-shadow: 0 1px 0 #fff inset, 0 24px 60px -28px rgba(0,0,0,0.22);
}
```

라이트 모드 전용 "Figma 프레임 카드" 디자인이 모든 콘텐츠를 감싸고 있었어요. 헤더 좌상단에 흰색 모서리가 살짝 비치면서 BrandLogo 영역에 박스처럼 보였던 거예요. 다크 모드는 `--do-surface`가 어두운 색이라 안 보였고요.

**교훈:** 범인은 컴포넌트 안이 아니라 **그 컴포넌트를 감싸는 wrapper에 있을 때**가 정말 많아요. 줌아웃 먼저 한 번 하고 들어가는 습관이 진짜 중요해요. 컴포넌트 → wrapper → layout 순서로 추적!

## 뻘짓 #4 — 캐시를 첫째 의심으로 안 둠

이건 클래식한 함정이에요.

제가 "옛 버전이 계속 보이는 것 같은데?"라고 했을 때 AI가 한 행동은 **`.next/cache/images` 파일만 삭제**. 그리고 "캐시 비웠으니 hard refresh 해주세요" 안내.

근데 Next.js 15 + Turbopack은 **메모리에 빌드 결과를 들고 있어요**. 파일을 지운다고 메모리가 비워지는 게 아니에요. **dev 서버 자체를 재시작했어야** 했어요.

이 단순한 조치를 가장 마지막에 한 거예요. 진단의 우선순위가 거꾸로였어요.

**교훈:** 파일이 맞는데 화면이 옛 버전을 보여주면, **90%는 캐시**예요. 그리고 캐시는 디스크 캐시만이 아니라 **메모리 캐시도 같이 의심**해야 해요. dev 재시작은 30초 걸리는 가장 싼 진단이니까 먼저 돌려보세요.

## 뻘짓 #5 — 사용자의 단순한 직관을 늦게 들음

이게 제일 뼈아팠어요.

제가 도중에 이렇게 짚어줬어요:

> "내 개인적인 생각엔 캐시 이슈가 맞을 것 같은데, 아까 계속 아예 다른 이미지 가져왔던 거 보면. 그냥 서버를 껐다 켜보면 안 돼?"

**그게 진짜 답이었어요.** 그런데 AI는 그 직전까지 픽셀 분석, CSS filter 실험, layout wrapper 추적 같은 **정밀 진단**으로 빠져들고 있었어요. 시간이 오래 걸린 정밀 진단보다, 사용자의 한 문장이 정답에 더 가까웠던 거예요.

**교훈:** AI는 분석 도구를 많이 가지고 있어서 **자꾸 정밀 진단으로 빠지는 경향**이 있어요. 그래서 사용자의 짧고 단순한 직관(특히 "그냥 X 해보면 안 돼?" 같은 한 줄)을 **첫 5분 안에 검증**하는 습관이 정말 중요해요. 그게 거의 정답이거든요.

## 5가지 뻘짓 한 장 요약

| # | 패턴 | 본질 | 진짜 답 |
|---|---|---|---|
| 1 | "다른 로고를 쓰고 있다" | Read 도구가 흰 배경 합성 | 도구 시각화는 사용자가 보는 것과 다름 |
| 2 | CSS filter·픽셀 변환 들이댐 | 광학 인지 ≠ 픽셀 데이터 | 광학 차이는 광학으로 풀어라 |
| 3 | BrandLogo만 들여다봄 | 줌인의 함정 | 컴포넌트 → wrapper → layout 줌아웃 |
| 4 | `.next/cache/images`만 지움 | 메모리 캐시 못 봄 | dev 재시작이 가장 싼 진단 |
| 5 | 정밀 진단에 빠짐 | 사용자 직관 무시 | 한 줄 의견부터 검증 |

## 오늘의 4가지 체크리스트

다음번엔 이거부터 돌릴 거예요. 여러분도 가져다 쓰세요:

- [ ] **도구의 시각화를 신뢰하지 말 것** — Read는 합성, 사용자 뷰어는 또 다른 합성. 둘이 다를 수 있음
- [ ] **줌아웃 먼저** — 컴포넌트 안에서 못 찾으면 wrapper, layout, body 순으로 추적
- [ ] **캐시는 1순위 의심 대상** — 파일이 맞는데 안 보이면 90% 캐시. 메모리 캐시까지 의심
- [ ] **사용자의 짧은 직관 = 거의 정답** — 정밀 진단에 빠지지 말고 한 줄 의견을 첫 5분 안에 검증

## 마무리 — 진짜 비싼 수업료

오늘 4시간은 진짜 비싼 수업료였어요. 그런데 끝나고 나니 좋은 패턴 다섯 개를 얻었으니까, **시간당 1패턴**이면 못 봐줄 단가는 아닌 것 같기도 하고요. ㅋㅋ

진짜 핵심은 이거예요:

> **AI 페어 프로그래밍은 "AI가 알아서 해주는 것"이 아니라 "인간이 옳은 방향을 짚어줄 때 빨라지는 것"이에요.**

AI가 정밀 진단 도구를 많이 가지고 있다고 해서, 그게 항상 정답으로 빨리 가는 길은 아니에요. 오히려 사용자(인간)의 단순한 직관이 더 빠른 길일 때가 많아요. 이걸 인정하고 처음부터 짧은 직관을 검증하는 습관을 들이면, 다음 4시간짜리 뻘짓을 30분짜리로 줄일 수 있을 거예요.
