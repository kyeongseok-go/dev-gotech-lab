# 작업 흐름 규칙

## 시작 전 필수 절차

1. 작업을 시작하기 전에 반드시 `docs/techspec.md`를 먼저 읽는다.
2. 그 다음 `docs/plan.md`를 읽고, 아직 완료되지 않은 가장 위 작업부터 진행한다.
3. 작업 범위가 불분명하면 코드를 수정하기 전에 먼저 현재 이해한 작업 범위를 5줄 이내로 요약한다.
4. 한 번에 하나의 기능 단위만 구현한다.

## 구현 원칙

- 한 번의 작업에서는 가능한 한 적은 파일만 수정한다.
- 대규모 리팩터링은 사용자 요청이 있을 때만 수행한다.
- 먼저 작은 단위의 동작 가능한 구현을 만들고, 그 다음에 다듬는다.
- 새로운 의존성을 추가할 때는 반드시 추가 이유를 한 줄로 남긴다.
- 데이터 스키마, 인증 방식, 배포 설정을 바꾸는 작업은 먼저 변경 요약을 제시한 후 진행한다.
- GitHub Issue 또는 plan.md 체크박스와 연결되는 단위로 작업한다.

## 테스트 및 검증

- 기능 구현 전 테스트가 쉬운 경우 테스트를 먼저 작성한다.
- 테스트를 바로 쓰기 어려운 UI 작업은 최소한 수동 검증 체크리스트를 함께 남긴다.
- 작업이 끝나면 아래 순서로 검증한다.
  1. 타입 체크
  2. 린트
  3. 관련 테스트
  4. 수동 확인 포인트 요약

## GitHub 규칙

- 기능 1개가 끝날 때마다 커밋한다.
- 커밋 메시지는 한국어 요약 한 줄 + 필요한 경우 상세 bullet 2~3개 형식으로 작성한다.
- Pull Request 설명에는 변경 요약 / 테스트 결과 / 스크린샷 여부 / 위험 요소를 남긴다.
- README와 변경 이력은 배포 가능한 수준의 기능이 생길 때마다 갱신한다.

## 응답 규칙

- 항상 다음 형식으로 답한다.
  1. 이해한 작업
  2. 수정/생성할 파일
  3. 구현 내용
  4. 검증 결과
  5. 다음 권장 작업
- 코드 생성 후에는 바로 붙여넣어 쓸 수 있도록 파일 경로를 명확히 표시한다.
