# 아키텍처 규칙

## 데이터 역할 분리

- 블로그/프로젝트 원문은 `content/` 아래 MDX 파일이 원본이다.
- D1은 공개 서비스 데이터, 캐시, 집계 데이터만 저장한다.
- R2는 이미지, 썸네일, OG 이미지, 첨부 파일, 3D 에셋의 단일 저장소다.
- Supabase는 인증, 댓글, 구독, 멤버십의 원본 데이터만 담당한다.
- 같은 원본 엔티티를 D1과 Supabase에 동시에 저장하지 않는다.
- 파일 저장소를 R2와 Supabase Storage로 이원화하지 않는다.

## 페이지 우선순위

1. 홈
2. 블로그 목록/상세
3. 프로젝트 목록/상세
4. About
5. Subscribe
6. Services 목록
7. AI Showcase 목록/상세
8. 실서비스 MVP 1개

## 1주차 범위 제한

- 1주차에는 뉴스 애그리게이터 수준의 서비스 1개만 MVP로 만든다.
- AI Showcase는 우선 목록형 카드와 상세 페이지 틀까지만 만든다.
- 3D 히어로는 선택 기능이다. 배포가 늦어지면 2D 폴백만 먼저 배포한다.
- 댓글, 로그인, 유료 구독, 실시간 기능은 구조만 준비하고 기본 구현은 Phase 2로 넘긴다.
- 관리자 페이지는 만들지 않는다.

## 구현 상세 규칙

- App Router 기준으로 구현한다.
- `src/` 디렉토리 구조를 사용한다.
- 공용 유틸은 `src/lib/`, 공용 UI는 `src/components/`, MDX 컴포넌트는 `src/components/mdx/`에 둔다.
- UI 라이브러리는 shadcn/ui를 기본으로 하고, Aceternity UI는 정말 필요한 컴포넌트만 가져온다.
- GitHub 링크, 데모 링크, 기술 스택, 스크린샷은 프로젝트/AI Showcase 카드의 핵심 필드다.

## 금지 사항

- Node middleware 전제를 깔고 인증 구조를 짜지 않는다.
- 1주차에 대형 파일 변환기, 가상 피팅, 복잡한 실시간 시스템을 구현하려고 하지 않는다.
- 홈과 소개 페이지를 과도한 인터랙션으로 무겁게 만들지 않는다.
